# Unified Persistence Tool Design

Date: 2026-02-01
Status: Draft

## Background

We currently persist user data in multiple places:
- `~/claudecodeimpact/data.db` (key-value JSON via Tauri commands)
- `~/.claudecodeimpact/claudecodeimpact/*` (runtime assets, scripts, legacy JSON)
- `~/.claudecodeimpact/docs/*` (user-managed documentation)
- Browser `localStorage` (fallback for web mode or legacy keys)
- Platform data dir (e.g., `~/.local/share/claudecodeimpact/command-stats.json`)

This spec unifies the persistence surface with a clear taxonomy and a single API while keeping large/binary/user-managed data out of the key-value store.

## What lives under ~/.claudecodeimpact

`~/.claudecodeimpact` is the **user-visible app workspace** for assets and caches that are not suitable for a JSON KV store:

- `~/.claudecodeimpact/claudecodeimpact/statusline/`
  - Statusline templates and backups (installed/apply/restore flows).
- `~/.claudecodeimpact/claudecodeimpact/statusbar/`
  - Statusbar scripts (GUI status bar integration).
- `~/.claudecodeimpact/claudecodeimpact/scrollback/`
  - Terminal scrollback buffers (binary, large, per-session).
- `~/.claudecodeimpact/claudecodeimpact/*.json` (legacy)
  - `disabled_env.json`, `custom_env_keys.json`, `disabled_hooks.json`, `statusbar-settings.json`.
  - These are migrated into `~/claudecodeimpact/data.db` on read and should be treated as legacy cache files.
- `~/.claudecodeimpact/docs/reference/`
  - User-provided documentation symlinks for the Knowledge feature.
- `~/.claudecodeimpact/docs/distill/`
  - Distilled/processed docs generated by the app.

Why this split:
- Scripts/docs/scrollback are **files** (large/binary/user-owned) and do not belong in JSON KV storage.
- `data.db` remains small, structured, and easy to migrate/backup.

## Goals

- Single persistence API for UI and settings across web and Tauri.
- Deterministic storage location for each data type.
- Safe migrations from legacy localStorage and JSON files.
- Clear separation between:
  - UI preferences (small JSON)
  - App settings (small JSON)
  - User assets (scripts/docs)
  - Runtime cache (scrollback, indexes)

## Non-goals

- Storing large binary assets or user files inside `data.db`.
- Replacing the OS keychain for secrets.

## Unified Persistence Tool (UPT)

### 1) Storage Taxonomy

- **KV Store (`data.db`)**
  - Small JSON values: preferences, settings, view modes, analytics opt-in.
  - Namespaced keys: `ui.*`, `settings.*`, `terminal.*`, `analytics.*`, `app.*`.
- **File Store (`~/.claudecodeimpact`)**
  - Scripts, docs, large assets.
  - Preserved for user visibility and portability.
- **Runtime Cache (platform data dir)**
  - Search indexes, command stats, transient caches.

### 2) Key Namespace

```
ui.<feature>.<key>
settings.<domain>.<key>
terminal.<key>
analytics.<key>
app.<key>
```

Rules:
- `ui.*` is front-end owned (Jotai, view modes, local UI state).
- `settings.*` is backend-owned settings merged into `~/.claude` config.
- Avoid flat keys; enforce namespaces to prevent collisions.

### 3) Data Lifecycle

- **Read path**
  1) Try `data.db` key.
  2) If missing, try legacy localStorage or legacy JSON file.
  3) If found in legacy, migrate into `data.db` and optionally clear legacy.
- **Write path**
  - Always write through the UPT API (JS -> Tauri -> data.db), with web fallback to localStorage.

### 4) API Design

#### Frontend (TypeScript)

```ts
export interface PersistenceAdapter {
  get<T>(key: string): Promise<T | undefined>;
  set<T>(key: string, value: T): Promise<void>;
  remove(key: string): Promise<void>;
  getMany(keys: string[]): Promise<Record<string, unknown>>;
}

export interface PreferenceDescriptor<T> {
  key: string;
  defaultValue: T;
  namespace: "ui" | "settings" | "terminal" | "analytics" | "app";
}
```

- Provide `createPreferenceAtom()` wrapper for Jotai.
- Provide `usePreference()` hook for non-atom usage.
- Provide `migrateLegacy()` for localStorage and file-based legacy keys.

#### Backend (Rust)

```rust
#[tauri::command]
fn get_data(key: String) -> Result<Option<Value>, String>;

#[tauri::command]
fn get_data_batch(keys: Vec<String>) -> Result<HashMap<String, Value>, String>;

#[tauri::command]
fn set_data(key: String, value: Value) -> Result<(), String>;

#[tauri::command]
fn remove_data(key: String) -> Result<(), String>;
```

- Enforce key namespace prefix validation in backend.
- Provide `migrate_legacy_*` helpers for JSON files under `~/.claudecodeimpact/claudecodeimpact/`.

### 5) Migration Strategy

- **Phase 1 (silent)**: read legacy keys and migrate on first access.
- **Phase 2 (cleanup)**: add a `clean_legacy_persistence` command or optional UI action.
- **Phase 3 (enforcement)**: remove all direct localStorage writes for Tauri runtime.

### 6) Error Handling

- All persistence failures must be **non-blocking** for UI.
- Return defaults on failure and log once per session.

### 7) Observability

- Add minimal logging on migration success/failure.
- Optional dev-only debug view to inspect `data.db` keys.

## Implementation Plan (Incremental)

1) Formalize `PersistenceAdapter` in `src/lib/persistence/`.
2) Wire Jotai storage to UPT (already partially done via `atomWithStorage`).
3) Migrate remaining direct localStorage usage (doc reader scroll, last path, etc.).
4) Add legacy JSON migration helpers for `~/.claudecodeimpact/claudecodeimpact/*.json` (if not already migrated).
5) Add optional cleanup command and UI toggle.

## Open Questions

- Should `data.db` live under `~/.claudecodeimpact` or `~/claudecodeimpact` long term?
- Should scrollback be capped/rotated based on total size across sessions?
- Should there be an export/import UI for `data.db` and docs?
